# プロジェクト概要

私はソフトウェアエンジニアで、ボクシングジムのマネージャーから小さな業務改善の相談を受けている。

現在、ジムでは試合のチケット販売を **LINEグループ** で案内している。

しかし、以下の問題がある。

## 現在の運用フロー

1. ジムのグループLINEで「試合チケットを販売します」と案内
2. チケット希望者は **マネージャーに個人LINEで連絡**
3. マネージャーが個別に注文内容を管理

このフローの理由は、

* 誰がチケットを買ったか / 買わないかを
* グループLINEの他のメンバーに知られたくない

という配慮のため。

しかしこの方法だと、

* 注文情報が **個別LINEに散らばる**
* マネージャーが **誰が何枚買ったかを管理しづらい**

という問題がある。

---

# 解決したいこと

以下を実現したい。

1. グループLINEでチケット販売の案内をする
2. 注文したい人は **URLから注文フォームに入力**
3. 注文情報は **1か所に集約される**
4. マネージャーは **一覧画面で管理できる**

また以下の条件がある。

* 他の会員には注文情報が見えない
* できるだけシンプル
* 一度作ったら **長期間放置できるシステム**
* 運用コストは最小
* 追加費用はできれば **0円**

---

# 技術的な前提

私はすでに **VPSを運用しており、別のアプリを動かしている**。

そのため、

* VPSの運用コスト
* インフラ管理

はすでに発生している。

よって、このシステムを **同じVPSに追加する場合の追加コストはほぼゼロ** と考えている。

またドメインについても、なるべくお金をかけない方法でいきたいと思ってる

---

# システムの要件（想定）

## 会員側

グループLINEで送られたURLを開き、以下を入力する。

例：

* 名前
* チケット種別
* 枚数
* 備考

送信すると注文が登録される。
ログインは不要。

---

## 管理者側

マネージャーが管理画面で確認できる。

例：

* 注文一覧
* チケット枚数集計
* 支払い状況
* チケット受け渡し状況
* CSV出力

---

# 重要な設計方針

以下を優先する。

### 1. 放置耐性

数か月触らなくても壊れない構成。

### 2. シンプルさ

外部サービス依存を増やさない。

### 3. 運用の軽さ

マネージャーが簡単に使える。

### 4. 低コスト

追加費用はできれば **ゼロ円**。

---

# 検討したアーキテクチャ

以下を比較検討した。

### 1. Supabase + Vercel

メリット

* 開発が簡単
* マネージドDB

デメリット

* 無料プランは1週間アクセスがないと停止
* VPSがすでにあるのでメリットが小さい

---

### 2. Googleフォーム + スプレッドシート

メリット

* 放置耐性が高い
* インフラ不要

デメリット

* UIが弱い
* 拡張性が低い

---

### 3. VPS上に小さいWebアプリ

メリット

* 追加費用ゼロ
* 既存運用に乗る
* 柔軟

デメリット

* 少し開発が必要

現時点では **この案が最有力**。
